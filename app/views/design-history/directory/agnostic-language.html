Agnostic language

We have changed approach to the language and design of the directory to ensure that we are utilising an agnostic language style. This covers more than the language used within the content of the design though and came from a co-design session that we ran during Sprint 3.

Risks and warnings

Prior to this approach being agreed, we had a section within the Directory design that had a few working titles but the aim of the function was always to inform users to potential hazardous situations where flags or alerts may be required.

It was concluded early on during a co-design session that this function not only wasn't requried within the service, but could also create damaging bias to potential user groups.

Negative bias

One of the key user needs for the directory design is "to provide a service with the information regarding touch points for a child and which professionals have been in contact with that child over time, in order for those professionals to be easily contacted". The key point here is to build enough of a picture without distorting the viewpoint for the end user.
