Agnostic language

We have changed approach to the language used and design approach of the directory to ensure that we are utilising an agnostic language style. This covers more than the language used within the content of the design though and came from a co-design session that we ran during Sprint 3.

Risks and warnings

Prior to this approach being agreed, we had a section within the Directory design that had a few working titles but the aim of the function was always to inform users to potential hazardous situations where flags or alerts may be required.

It was concluded early on during a co-design session that this function not only wasn't requried within the service, but could also create damaging bias to potential user groups.

Negative bias

One of the key user needs for the directory design is "to provide a service with the information regarding touch points for a child and which professionals have been in contact with that child over time, in order for those professionals to be easily contacted". The key point here is to build enough of a picture without distorting the viewpoint for the end user.

During the co-design session we talked about the negative connotations that could come out of including a risks section to the user interface, how it would be tricky to include what would undoubtedly be a subjective set of data.

Context check

To ensure that removing information from the design wouldn't remove knowldge that would be requied by the professionals, we taked through a number of potential scenarios to ensure that if there were any situations where professionals may need additional information that they would be able to find the correct level sufficiently. On each occasion it was agreed that the professional timeline followed up with conversations with professionals within the timeline would allow for the relevant information to be surfaced.

Additional changes

Standing assumptions

at this point in alpha we're still yet to place the designs in front of users, so there are a number of high level assumptions connected with the article and choices made as laid out above:

  • the level of detail is sufficient for users
  • it is possible to ascertain potential sitations from the selected timeline period
  • users will wish to view only a certain part of the timeline, but it is currently unknown if filters on time or type of contacts are required
